## Executive Summary

Azumi is an ambitious, well-architected full-stack Rust web framework with a distinctive thesis: **compile-time validation of everything that can be validated**, zero-hydration architecture, and AI-first design. The codebase demonstrates exceptional security consciousness, extensive test coverage (43% of total LOC is tests), and thoughtful macro engineering.

**Verdict:** Production-ready for its target use cases. Strong architectural foundation with clear growth paths.

## 1. Architecture & Design

### 1.1 Core Thesis — Zero Hydration, HTML as Truth

Azumi's architectural bet is that modern web development over-engineers frontend interactivity. By sending HTML over the wire and using a ~3KB client runtime (event delegation + Idiomorph DOM morphing), it eliminates:
- Virtual DOM overhead
- WASM download tax (~150KB+ in Leptos/Dioxus)
- Hydration complexity
- JS/TS type-system fragility

**Assessment:** This is a defensible, increasingly popular architecture (HTMX-adjacent). The framework executes this vision consistently.

### 1.2 Compile-Time Validation Pipeline

The `html!` macro runs a sophisticated 7-stage validator:

```
1. CSS Validation       → property names, syntax
2. Node Order           → Script → Content → Style ordering
3. Raw Usage Check      → blocks ALL Raw() unconditionally
4. Format Detection     → blocks format! with web-content patterns
5. Class/ID Validation  → classes must be defined in <style> blocks
6. HTML Structure Rules → tables, lists, forms, buttons, headings
7. Attribute Validation → valid HTML5, data-*, aria-*, on:* events
```

**Strengths:**
- Unconditional `Raw()` ban is the right security choice
- `format!` detection prevents the #1 AI-generated XSS vector
- Class/ID co-validation is genuinely innovative (industry-first claim holds)

**Concerns:**
- Validation is **short-circuiting** — first error stops the pipeline. Users see one error at a time, which can be frustrating for large refactors.
- The 7-stage pipeline is embedded in `macros/src/lib.rs` and `validators.rs`. Adding new validators requires understanding the entire chain.

### 1.3 Component System

The `#[azumi::component]` macro transforms functions into modules with:
- `Props` struct + `PropsBuilder` (via `typed-builder` pattern)
- Zero-clone rendering via move semantics
- Automatic `az-scope` wrapping for live state components

**Strengths:**
- Builder pattern eliminates manual parameter passing
- Move semantics prevent accidental clones
- Live state auto-detection (first param named `state` + reference to non-primitive) is elegant

**Concerns:**
- The `state` parameter naming convention is implicit magic. A user naming their param `s` or `ctx` won't get live wrapping.
- `Props::builder().build().expect("...")` is verbose. A `try_build()` or infallible builder for all-required props would be ergonomic.

### 1.4 Live State Architecture

Live state uses HMAC-signed JSON embedded in DOM attributes:
```html
<div az-scope="signed_json" az-struct="Counter" az-predictions="[...]">
```

- State is signed server-side with timestamp + HMAC-SHA256
- Predictions (optimistic UI) are auto-generated from `#[azumi::live]` structs
- Client runtime verifies signatures before applying updates

**Strengths:**
- HMAC prevents state tampering without server secrets
- Timestamp prevents replay attacks
- User-scoped signing (`sign_state_for_user`) prevents cross-user replay
- Optimistic UI with automatic rollback on failure

**Concerns:**
- State size grows with struct complexity. Large states could exceed DOM attribute limits.
- The `panic!` in `to_scope()` on serialization failure is harsh — could fail at runtime for complex types.
- One-hour max state age (`MAX_STATE_AGE_SECS = 3600`) may be too short for some use cases (e.g., long forms). No configurability.

## 2. Security Analysis

### 2.1 XSS Prevention — Excellent

| Layer | Mechanism | Status |
|-------|-----------|--------|
| Text interpolation | Auto-escaped via `Escaped<T>` | ✅ |
| `Raw()` | Unconditionally banned in `html!` | ✅ |
| `format!` | Blocked when combined with web patterns | ✅ |
| `<script>{var}</script>` | Auto-escapes `</script>` (case-insensitive) | ✅ |
| `<style>{var}</style>` | Auto-escapes `</style>` (case-insensitive) | ✅ |
| `json_data!` | Serde serialize + `</script>` escape | ✅ |
| SEO helpers | `escape_html()` for meta content | ✅ |
| JS string escaping | `escape_js_string()` for action handlers | ✅ |

**Security Audit Finding:** The framework's security posture is **strong**. All major XSS vectors are covered. The unconditional `Raw()` ban is particularly admirable — most frameworks provide escape hatches; Azumi correctly treats them as footguns.

### 2.2 HMAC State Signing — Excellent

```rust
// Format: "{json}|{timestamp}|{signature_base64}"
// User-scoped: "azu:{user_id}:{json}|{timestamp}|{signature}"
```

- Secret from `AZUMI_SECRET` env var (min 32 bytes in release)
- Dev fallback with loud warnings
- `TooManyPipes` limit (10) prevents DoS on parsing
- `rfind('|')` approach is robust against JSON containing `|`

**Minor Concern:** In debug builds, `AZUMI_DEV_MODE` allows the default secret. The warning is printed to stderr but could be missed in some environments. Consider making this more prominent or requiring explicit opt-in.

### 2.3 CSP Support — Good

- `CspNonce` generates 128-bit random nonces (16 bytes base64)
- Axum middleware for per-request nonce injection
- `ContentSecurityPolicy` builder with Azumi defaults
- Default policy removes `'unsafe-inline'` from `script-src` when using nonces

**Concern:** The default `style-src` includes `'unsafe-inline'` because scoped CSS uses inline `<style>` tags. The docs acknowledge this tradeoff but there's no built-in nonce injection for `<style>` tags in `html!`. Users must manually add `nonce={nonce.as_str()}` to every `<style>`.

### 2.4 CSRF — Acknowledged Gap

The framework **correctly** does NOT implement CSRF protection, with clear documentation explaining why:
- Actions are JSON APIs designed for JS client calls
- Auth is application responsibility
- Recommends SameSite cookies, double-submit cookies, or custom middleware

**Assessment:** This is the right design choice. CSRF protection belongs at the auth layer, not the framework layer.

### 3.2 Macro Engineering

The proc-macro crate (~8,800 LOC) is the project's crown jewel:

**Strengths:**
- Clean separation: `token_parser.rs` (AST), `codegen.rs` (generation), `validators.rs` (validation)
- `proc_macro2` used throughout for span preservation
- Error messages use `quote_spanned!` for precise diagnostics
- `trybuild` UI tests for compile-fail scenarios (15 test cases)

**Concerns:**
- `token_parser.rs` at 1,349 lines is large. The `Parse` implementations for `Node`, `Element`, `Attribute`, etc. are deeply nested.
- Some parser logic uses ad-hoc string scanning (e.g., `find("<style")` in devtools) which could break with macro nesting.
- `style.rs` (1,202 lines) handles both CSS DSL parsing AND scoping. Consider splitting.

### 3.3 Error Handling

- `thiserror` used for custom error types (`VerifyStateError`)
- `Result` propagation in most places
- Some `expect()`/`panic!` in macro-generated code for truly exceptional cases

**Concern:** The `LiveState::to_scope()` panics on serialization failure. In a production render path, this could crash the server. Consider returning a `Result` and letting the caller handle it gracefully.

### 3.4 Unsafe Code

**No `unsafe` blocks found** in core library or macros. This is commendable for a web framework.

## 6. Client Runtime

### 6.1 Azumi.js (~3KB)

**Strengths:**
- Event delegation for `az-on` attributes (efficient, no per-element listeners)
- Idiomorph integration for DOM morphing (preserves focus, input state)
- Optimistic UI with automatic rollback
- Local state restoration after morphing
- Concurrent update prevention (signed state can't pipeline)
- Hot reload WebSocket with graceful degradation

**Concerns:**
- Single-file class (`class Azumi`) is 600+ lines. Could be split for maintainability.
- Error handling uses `console.error` — no user-facing error UI for action failures
- No TypeScript definitions (framework is Rust-first, but TS defs would help hybrid teams)
- `idiomorph.js` is vendored (~2.5KB). No version pin or checksum verification.

### 6.2 Protocol Design

The server protocol is simple and robust:
- Actions return HTML fragments
- Live updates return full scope HTML
- State is signed and verified

**Concern:** The WebSocket endpoint (`/_azumi/live_reload`) for hot reload is enabled in dev mode. In production, if `devtools` feature is accidentally enabled, this exposes a WebSocket. Consider a compile-time or runtime guard.

## 7. Potential Issues & Recommendations

### 7.1 High Priority

| Issue | Location | Recommendation |
|-------|----------|----------------|
| Live state `panic!` on serialize failure | `macros/src/live.rs`, `src/security.rs` | Return `Result` and let caller handle gracefully |
| No route type safety | `docs/guide.md` (design doc exists) | Implement `routes!` macro per `DESIGN_ROUTE_SAFETY.md` |
| `style-src 'unsafe-inline'` default | `src/csp.rs` | Document nonce-based style CSP pattern more prominently |
| Hot reload WS exposed if `devtools` enabled | `src/hot_reload.rs` | Add runtime check for debug builds only |

### 7.2 Medium Priority

| Issue | Location | Recommendation |
|-------|----------|----------------|
| `state` param naming magic | `macros/src/component.rs` | Allow `#[live_state]` attribute for explicit opt-in |
| One-hour max state age | `src/security.rs` | Make `MAX_STATE_AGE_SECS` configurable via env var |
| Token parser monolith | `macros/src/token_parser.rs` | Split into submodules (element.rs, attribute.rs, expression.rs) |
| No `no_std` support | `src/lib.rs` | Document as non-goal or add `std` feature gate |
| Missing aria-* value validation | `macros/src/accessibility_validator.rs` | Validate `aria-expanded="true"` not `aria-expanded="yes"` |

### 7.3 Low Priority

| Issue | Location | Recommendation |
|-------|----------|----------------|
| `Props::builder().build().expect()` verbosity | Generated code | Add infallible `build_unchecked()` for all-required props |
| No TypeScript definitions | `client/` | Add `.d.ts` for hybrid teams |
| Vendored idiomorph without checksum | `client/idiomorph.js` | Add version comment and SHA256 checksum verification |
| Benchmarks not in CI | `Cargo.toml` | Add criterion regression to CI |

## 8. Strengths (What Azumi Does Exceptionally Well)

1. **Security-First Design**: Unconditional `Raw()` ban, `format!` detection, HMAC-signed state, comprehensive escaping. This is the most security-conscious Rust web framework I've reviewed.

2. **AI-Ready**: `AGENTS.md`, `AZUMI_RULES`, compile-time anti-pattern detection, and the validation pipeline are designed for AI-generated code. This is forward-thinking.

3. **Test Coverage**: 43% test LOC ratio with property-based testing, stress tests, and UI tests. Exceptional.

4. **Zero-Hydration Architecture**: The ~3KB runtime + HTML-over-the-wire approach eliminates an entire class of frontend complexity.

5. **CSS-HTML Co-Validation**: The class/ID validation that cross-references `<style>` blocks is genuinely innovative and catches real bugs.

6. **Documentation Quality**: Every major design decision has an RFC-quality document. The comparison matrix, migration guides, and AI guide set a high bar.

7. **Compile-Time Errors Over Runtime Errors**: The framework consistently chooses to fail at compile time rather than runtime. This aligns with Rust's philosophy.
